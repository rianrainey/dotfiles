# Principal Lens

Use this reference when the user wants to grow from staff toward principal-level judgment.

Focus explanations through these lenses:

## Business Leverage

Explain:
- which flows matter most to revenue, margin, risk, compliance, or user trust
- where the code reflects business priorities
- which parts of the system are critical versus incidental

## Architecture

Explain:
- system boundaries
- dependency direction
- coupling hotspots
- stable interfaces versus historical accidents
- where changes will have wide blast radius

## Operations And Reliability

Explain:
- runtime behavior
- jobs and async work
- external dependencies
- failure modes
- observability and support surfaces

## Product And Execution Strategy

Explain:
- where a newcomer can move fast safely
- where the team likely pays coordination costs
- what should be standardized versus left flexible
- where the biggest simplification opportunities may be

## Leadership Judgment

Encourage the user to ask:
- What does the business most need from this system?
- Where does complexity compound?
- Which abstractions are helping, and which are legacy drag?
- What would I protect if I owned this system for three years?
- Where would I invest to reduce risk or increase delivery speed?
